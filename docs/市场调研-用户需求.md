# AI 编程场景下的语音输入 — 用户需求调研

> 调研时间：2026-03-24
> 来源：Reddit (r/ClaudeAI, r/vibecoding, r/ClaudeCode), Hacker News, V2EX, 竞品官网

---

## 一、AI 编程工具的语音支持现状

| 工具 | 语音支持情况 | 备注 |
|------|-------------|------|
| Claude Code | 2026年3月推出 `/voice` 命令 | 识别准确率不高，把"Claude Code"识别成"clot coat" |
| Cursor | 聊天框有麦克风图标 | 基础功能，无开发者专属优化 |
| VS Code | 内置语音支持（Win+H 或扩展） | 通用语音输入，不理解编程术语 |
| GitHub Copilot | Copilot Voice（实验性） | 仅限语音写代码，非 prompt 输入场景 |

## 二、程序员核心诉求（按出现频率排序）

### 1. 减少打字量，保持心流状态
> "I'm an AI Dev who got tired of typing 3,000+ words/day to Claude"
> "I'm mainly using AI at all because typing is soooo slow"

程序员每天给 AI 写大量 prompt，打字是最大的摩擦点。

### 2. 技术术语的准确识别
> "Claude Code's /voice command heard 'clot coat' when I said 'Claude Code.'"

**最大痛点**：通用语音识别无法处理 API 名称、库名、缩写、驼峰命名等。

### 3. 隐私和本地运行
> "Why I Cancelled My Wispr Flow Subscription" -- 开发者担心代码相关语音数据被上传

处理私有代码库的开发者对云端语音服务有顾虑。

### 4. 低延迟
延迟超过 500ms 会明显打断思路。

### 5. Windows 支持严重不足
最好的工具都是 Mac only，**Windows 开发者是被严重忽视的群体**。

### 6. 价格敏感
> "Wispr flow and all these voice-typing solutions are damn expensive"

$144/年的订阅费让很多开发者望而却步。

---

## 三、现有工具未满足的需求

1. 没有"AI 编程场景"的专属模式
2. 无法自动区分"说给 AI 的指令"和"普通文本"
3. 自定义词汇表不够智能（不能从项目代码自动学习）
4. 中英混合输入识别率差
5. CLI/终端场景缺乏原生支持
6. 按住说话 vs 自动检测只支持一种模式

---

## 四、差异化方向建议

### Tier 1: 今晚就能做

1. **从项目代码自动提取词汇表** — 扫描 import/requirements.txt/package.json，自动生成词汇表
2. **AI Prompt 模式** — 将口语化开发指令整理为结构化 AI prompt
3. **连续录音模式** — 短按开始/暂停/继续，双击结束，适合写复杂 prompt
4. **识别结果预览确认** — 确认后再粘贴，可快速编辑

### Tier 2: 一两天能做

5. **应用感知自动切换模式** — 检测当前窗口，终端用 Prompt 模式，浏览器用通用模式
6. **中英文混合输入优化** — 针对编程场景优化 Whisper prompt
7. **快捷短语/语音模板** — 说"审查模式"自动添加 prompt 前缀

### Tier 3: 需要更多时间

8. **本地 Whisper 模型支持** — 集成 faster-whisper 作为可选离线后端
9. **Claude Code MCP Server 集成** — 注册为 Claude Code 的 MCP Server
10. **语音历史和常用指令** — 记录历史，推荐高频指令为快捷短语

---

## 五、关键结论

- **最大市场空白**: Windows 平台 + AI 编程场景 + 免费 = 几乎没有竞品
- **最强差异化**: 不是"语音转文字"本身，而是**"语音转高质量 AI prompt"**
- **最快见效**: 自动词汇提取 + 预览确认（解决术语识别错误和误粘贴两个最高频痛点）
- **竞品弱点**: Wispr Flow 太贵且 Windows 差；SuperWhisper 只支持 Mac；Claude Code /voice 识别准确率被吐槽
